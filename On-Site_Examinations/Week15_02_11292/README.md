### 題目：[UVa 11292 - Dragon of Loowate](https://onlinejudge.org/external/112/11292.pdf)

#### 解題思路
龍有 $n$ 個頭，每個頭有不同的直徑。你需要僱用 $m$ 個騎士，每個騎士有能砍斷的頭之最大直徑限制，且僱用成本等於該騎士的體力值
- 目標：用最少的金幣擊敗所有龍頭。若無法擊敗，則輸出 Loowater is doomed!
- 核心特性：為了節省金幣，對於每個龍頭，我們應該選擇「能砍下該頭的騎士中，體力最小的」那一位
- 核心策略：貪心法。將龍頭和騎士的直徑皆由小到大排序，利用雙指標依序匹配

#### 關鍵邏輯：
1. Sorting：
    - 將龍頭 (head) 與騎士 (knight) 的能力值分別從小到大排序
2. Two-Pointer Matching:
    - 用指標 i 指向龍頭，j 指向騎士
    - 若 knight[j] >= head[i]，表示該騎士足以砍下此頭，且因為已排序，該騎士是當前能砍下此頭的最廉價選擇，故僱用之 (ans += knight[j]) 並移向下一組匹配 (i++, j++)
    - 若 knight[j] < head[i]，表示該騎士太弱了，無法砍下此頭，直接捨棄該騎士，試下一個較強的 (j++)
3. 可行性驗證：
    - 若所有龍頭皆被匹配 (cnt == n)，輸出總費用；否則宣告任務失敗

#### 模擬運算(以龍頭 {5, 4}, 騎士 {7, 8, 4}為例)：
1. 排序後：龍頭 {4, 5}，騎士 {4, 7, 8}
2. 匹配開始：
    - head[0]=4, knight[0]=4：匹配成功，ans = 4，i=1, j=1, cnt=1
    - head[1]=5, knight[1]=7：匹配成功，ans = 4+7 = 11，i=2, j=2, cnt=2
3. 結果：完成所有匹配，輸出 11
- Time Complexity：$O(n \log n + m \log m)$，主要耗時在排序
- Space Complexity：$O(n + m)$，用於儲存龍頭與騎士資料
#### 語言差異與踩坑紀錄
* **學到了什麼：**
    - 貪心法的選擇：在此類配對問題中，排序是貪心策略生效的前提。沒有排序，我們無法確定當前騎士是否為「能完成任務的最廉價騎士」
    - 邊界判斷：題目未明確限制騎士數量一定要大於龍頭，故開頭 if (n > m) 的檢查是必要優化
    - 指標移動邏輯：注意在 if 與 else 分支中，指標的移動必須謹慎，確保不會遺漏任何可行的匹配可能
#### 總結 (Takeaway)
- 最小代價匹配問題 SOP：
    1. 將兩組需求與資源分別由小到大排序
    2. 使用雙指標從最小開始嘗試匹配
    3. 若資源滿足需求，立即匹配並累積成本
    4. 若資源不足，則捨棄該資源，尋找更強大的
