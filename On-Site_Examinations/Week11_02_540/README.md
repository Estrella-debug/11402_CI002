### 題目：[UVa 540 - Team Queue](https://onlinejudge.org/index.php?option=com_onlinejudge&Itemid=8&category=7&page=show_problem&problem=481)

#### 解題思路
在一個排隊系統中，人們不只是單獨的個體，還屬於不同的「團隊」
- 排隊規則：
    1. 插隊權利：如果一個人進入隊伍時，發現已經有同隊的隊友在排隊，他可以直接插隊到所有隊友的最後面，而不是整條隊伍的最後面
    2. 正常排隊：如果隊伍中沒有任何隊友，他才乖乖排在整條隊伍的最後面
- 目標：模擬 ENQUEUE（進隊）、DEQUEUE（出隊）與 STOP 指令，並輸出出隊的成員編號
- 核心策略：使用 「佇列中的佇列」。用一個大佇列 q 來管理「團隊的順序」，再用一組小佇列 team[] 來管理「每個團隊內部的成員順序」

#### 關鍵邏輯：
1. 團隊歸屬地圖 (id[])：
    - 使用一個大陣列（或雜湊表）將每個成員編號 a 映射到所屬的團隊編號 i。這能讓我們在 $O(1)$ 時間內判斷誰是隊友
2. ENQUEUE：
    - 當成員 a 要進隊時，先檢查他所屬團隊的小佇列 team[id[a]] 是否為空
    - 若為空：代表大隊伍中目前沒隊友，必須在大佇列 q 中登記該團隊進入排隊
    - 放入成員：不論是否為空，都將成員 a 放入對應的小佇列 team[id[a]] 中
3. DEQUEUE：
    - 先看大佇列 q 的前端，得知現在輪到哪個團隊 now 出人
    - 從該團隊的小佇列 team[now] 彈出最前面的成員
    - 團隊離場：如果彈出後該團隊的小佇列變空了，代表這團人都領完餐了，將團隊編號從大佇列 q 中移除
4. 清理重置：
    - 每組測試資料結束後，必須清空大佇列與所有小佇列，避免干擾下一組 Scenario

#### 模擬運算(團隊 1:[10, 11], 團隊 2:[20]為例)：
- ENQUEUE 10：team[1] 之前為空，q = [1], team[1] = [10]
- ENQUEUE 20：team[2] 之前為空，q = [1, 2], team[2] = [20]
- ENQUEUE 11：team[1] 之前不為空，直接排在隊友後。q = [1, 2], team[1] = [10, 11]
- DEQUEUE：q.front() 是 1，從 team[1] 彈出 10
- DEQUEUE：q.front() 仍是 1，從 team[1] 彈出 11。此時 team[1] 空了，q 彈出 1，變為 [2]
- Time Complexity：$O(1)$ 每項操作。查找團隊、進隊、出隊均為常數時間
- Space Complexity：$O(M)$，其中 $M$ 是所有團隊成員的總數

#### 語言差異與踩坑紀錄
* **學到了什麼：**
    - 結構化思考：學會將複雜的排隊問題拆解為「組間順序」與「組內順序」兩個層次
    - 全域大陣列：題目中成員編號可能高達 $1,000,000$，使用全域陣列 id[1000005] 處理映射比使用 std::map 更快且不會 TLE
    - 清空佇列：std::queue 沒有 .clear() 函式，學會了使用 while(!q.empty()) q.pop() 的標準清空方式
#### 總結 (Takeaway)
- 團隊/層次化排隊問題 SOP：
    1. 建立映射表紀錄成員與組別的關係
    2. 準備一個大佇列管理組別順序
    3. 準備一組小佇列管理各組內部成員
    4. 進隊時：若該組不在場，大佇列補位；小佇列入員
    5. 出隊時：從小佇列出員；若組別空了，大佇列退位
